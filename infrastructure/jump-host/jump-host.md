# Jump host: unseen-gate

## Device description

The jump host is the sole entry point into the UU P&L environment. It sits on
the public network and is dual-homed into the enterprise zone, making it the
attacker's first foothold and the only route inward from the outside.

The machine is called `unseen-gate`. It is not a target — it is a starting
position. It has no vulnerabilities, no interesting files, no credentials for
internal systems. Its purpose is to be a clean SSH landing pad from which
participants explore what lies beyond.

Five adversary accounts exist: `moist`, `teatime`, `carrot`, `angua`, `vimes`.
Each account is issued to one participant before the exercise begins. Access is
key-only. The accounts share the same environment and can see each other's
activity if they look.

From `unseen-gate`, participants can reach the enterprise zone directly. The
README in their home directory tells them the enterprise subnet and the two
known hosts. Everything else they find themselves.

## Container behaviour

The container exposes one service:

* SSH on port 22: key-only authentication, no password fallback

Five accounts with identical permissions, no sudo, no root access. The
container is minimal: Debian base, SSH server, basic network tools. There is
nothing to exploit on this box.

At startup, the entrypoint reads `/run/adversary-keys` (a bind-mounted file)
and distributes public keys to the appropriate account's `authorized_keys`.
It also copies `/run/adversary-readme.txt` to each home directory as `README`.

Both files are generated at deploy time and are not part of the image.

## Network position

```
Attacker
  │ SSH (key auth, port 22 on Hetzner / 2222 locally)
  ▼
unseen-gate
  ics_internet:   10.10.0.5   (public-facing)
  ics_enterprise: 10.10.1.5   (enterprise zone)
```

From `unseen-gate`, participants can reach `10.10.1.0/24` directly. They
cannot reach the operational zone (`10.10.2.0/24`), the control zone
(`10.10.3.0/24`), or the WAN (`10.10.4.0/24`) from here. The firewall
enforces this. Deeper access requires pivoting through enterprise zone hosts.

## Operator setup

Two files must exist before the container starts. Neither is committed to the
repository.

### `infrastructure/jump-host/adversary-keys`

One line per participant. Format: `username pubkey [comment]`

```
moist   ssh-ed25519 AAAA... participant1@team
teatime ssh-ed25519 AAAA... participant2@team
carrot  ssh-ed25519 AAAA... participant3@team
angua   ssh-ed25519 AAAA... participant4@team
vimes   ssh-ed25519 AAAA... participant5@team
```

Copy `adversary-keys.example` and fill in real public keys. The file is
gitignored. Do not commit real keys.

### `infrastructure/jump-host/adversary-readme.txt`

Generated by `orchestrator/generate.py` from `ctf-config.yaml`. Contains the
enterprise network map shown to participants on login. Run `make generate`
before deploying.

## Deployment

### Local (development / testing)

`ctf-config.yaml` sets `ssh_host_port: 2222` by default, so the container
binds `2222:22`. This avoids conflict with the host sshd which normally holds
port 22.

```bash
# Create adversary-keys with your own public key for one account
echo "moist $(cat ~/.ssh/id_ed25519.pub)" \
    > infrastructure/jump-host/adversary-keys

# Build and start everything
make generate
make build
make deploy

# Connect as if you were a participant (port-mapped route)
ssh moist@localhost -p 2222

# Alternatively, connect via the Docker bridge IP directly (no port flag needed)
ssh moist@10.10.0.5
```

### Hetzner (production)

```bash
# 1. Provision a Debian/Ubuntu instance on Hetzner
# 2. Move host sshd to port 2222 so the container can claim port 22
bash infrastructure/jump-host/setup.sh

# 3. Set ssh_host_port: 22 in orchestrator/ctf-config.yaml (change from 2222)

# 4. Fill in adversary-keys
cp infrastructure/jump-host/adversary-keys.example \
   infrastructure/jump-host/adversary-keys
# edit adversary-keys — add real participant public keys

# 5. Deploy
make generate && make build && make build-jump-host && make deploy
```

Participants then connect to:

```
ssh <username>@<hetzner-ip>
```

## Role in the simulator

The jump host is the gate. Its role is minimal by design:

* one SSH connection in
* one dual-homed interface out into the enterprise zone
* a README that orients without guiding

Everything interesting is behind it. The jump host itself offers nothing to
exploit. Participants who spend time on it are wasting time.

```
participant key pair
        ↓
ssh <username>@unseen-gate (10.10.0.5)
        ↓
read README — enterprise subnet 10.10.1.0/24, two known hosts
        ↓
nmap 10.10.1.0/24 or connect directly to 10.10.1.10 / 10.10.1.20
        ↓
enterprise zone
```
